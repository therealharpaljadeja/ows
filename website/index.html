OWS

Open Wallet Standard

An open, neutral standard for secure local wallet storage and agent access. One vault, one interface, every chain. Agents sign transactions without ever touching private keys. OWS exists to give every AI agent a wallet it can trust—and every user a wallet they control.

Design Principles

Built for the agent era

Five principles that make OWS different from every wallet SDK, cloud API, and ad-hoc keystore that came before it.

01

Keys never leave the vault

Private keys are encrypted at rest and decrypted only inside an isolated signing process. They are never exposed to agent prompts, LLM contexts, or parent processes.

02

Chain-agnostic by default

CAIP-2 chain identifiers and CAIP-10 account identifiers. EVM, Solana, Tron, Cosmos, Bitcoin, TON—all first-class citizens. No chain is special.

03

Policy before signing

Every agent transaction passes through a policy engine before key material is touched. Policies are attached to API keys, scoping what each agent can do. The owner bypasses policies entirely—sudo access to all wallets.

04

One interface, any consumer

CLI tools, MCP servers, REST APIs, and library imports all use the same abstract interface. A wallet created by one tool is usable by any other.

05

Open and forkable

Permissionlessly extensible. New chains, policy types, and storage backends are added via plugins, not core spec changes. CC0 licensed. No vendor lock-in.

Request

Agent calls ows_sign via MCP, REST, or SDK with a chain ID and transaction object.

Spending limits, allowlists, chain restrictions, and simulation requirements are evaluated before any key is touched.

Step 3

Sign

Key is decrypted in the isolated enclave, transaction signed, key wiped from memory. Signed transaction returned to caller.

→
Step 4

Submit (optional)

If RPC URLs are configured, the signed transaction is broadcast on-chain and the transaction hash is returned.

Why OWS

Zero compromise wallet access

🔒

Zero key exposure

Private keys are decrypted only inside an isolated subprocess. Agents, LLMs, and parent processes never see raw key material. Following the Privy/Turnkey enclave model—locally.

⛓

Zero chain lock-in

CAIP-2 chain identifiers work for every chain. EVM, Solana, Cosmos, Bitcoin, Tron, TON—and any future chain via the plugin system. Same interface, same vault, same policies.

🛡

Zero trust required

No cloud accounts, no API keys, no network dependency. Everything runs on your machine. Wallets are encrypted with scrypt + AES-256-GCM and stored at ~/.ows/.

⚙

Zero integration friction

MCP server for AI agents (Claude, GPT, LangChain). REST API on localhost for scripts. Library SDK for embedding. All three share one implementation.

📜

Zero unaudited actions

Every signing operation is logged to an append-only audit trail. Policy evaluations, simulation results, transaction hashes—all recorded in ~/.ows/logs/audit.jsonl.

🔌

Zero vendor lock-in

Built on open standards: Keystore v3, BIP-39/44, CAIP-2/10, MCP. Import from MetaMask, export to any wallet. CC0 licensed—use it, fork it, extend it.

Get Started

From zero to signing in 60 seconds

Choose your path: install the CLI for terminal use, or add the npm/pip package directly to your project. The language bindings are fully self-contained — they embed the Rust core via native FFI. No CLI required.

CLI 
curl -fsSL https://openwallet.sh/install.sh | bash
Node.js 
npm install @open-wallet-standard/core
Python 
pip install open-wallet-standard
Node.js — npm 
import { createWallet, signMessage }
  from "@open-wallet-standard/core";

const wallet = createWallet("agent-treasury");
// => accounts for EVM, Solana, BTC, Cosmos, Tron, TON

const sig = signMessage(
  "agent-treasury", "evm", "hello"
);
console.log(sig.signature);
Python — PyPI 
from open_wallet_standard import (
    create_wallet, sign_message
)

wallet = create_wallet("agent-treasury")
# => accounts for EVM, Solana, BTC, Cosmos, Tron, TON

sig = sign_message(
    "agent-treasury", "evm", "hello"
)
print(sig["signature"])

Specification

Modular by design

Seven sub-specifications, each covering a distinct concern. Add a new chain without touching the signing interface. Change the policy engine without touching storage.

Spec 01

Storage Format

Encrypted keystore format, vault directory structure, filesystem permissions, and audit logging.
Spec 02

Signing Interface

The core sign, signAndSend, signMessage, and simulate operations with error handling.
Spec 03

Policy Engine

Spending limits, allowlists, denylists, simulation requirements, time restrictions, and custom rules.
Spec 04

Agent Access Layer

MCP server tools, REST API endpoints, library SDK, and authentication for agent access.
Spec 05

Key Isolation

Process-level isolation, enclave protocol, passphrase handling, threat model, and defense in depth.
Spec 06

Wallet Lifecycle

Creation, import, export, backup, recovery, key rotation, and wallet discovery.
Spec 07

Supported Chains

Chain families, CAIP identifiers, HD derivation, RPC endpoints, and asset identification.
